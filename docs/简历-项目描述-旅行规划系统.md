# 旅行规划系统 - 简历项目描述（结合 design.md 与实现）

> 依据 `design.md` 工程设计文档与当前代码实现整理，区分「已实现」与「设计/目标态」，避免夸大。业务场景与术语与项目一致。

## 一、项目背景与定位（来自 design 1.1 / 1.2）

- **系统目标**：从零设计并实现「旅行规划智能体 TravelMind」，采用前后端分离架构，前端负责可视化与交互，后端负责编排、检索、证据聚合与结构化输出。
- **产品形态**：面向自由行，支持多轮对话：会话内澄清追问（多轮补充/resume 续答/reset）、会话态持久化、意图路由（create/edit/qa/reset），输出可编辑的结构化行程（itinerary v1）并支持连续优化（含 Edit Day N）；输出真实证据用于推荐解释（只读），不做交易闭环（不下单/支付/票务履约）。
- **MVP 边界**：支持首次生成、Edit Day N、证据展示、JSON/Markdown 导出；支持预算分项、风险分级、备选项、修订链与 diff；默认走标准检索模式。不默认启用 DeepResearch 等高成本增强路径。

## 二、推荐版本（一段式，可直接粘贴简历）

**旅行规划系统（TravelMind）**  
从零设计并实现的自由行行程规划系统，支持多轮对话：用户自然语言输入需求后，由 QP（意图识别 + 约束抽取）判断是否缺目的地/天数/预算等硬门槛，缺失则进入澄清分支，澄清服务维护 pending 状态，用户可多轮补充（通过 resume 续答）直至约束补齐，或 reset 清空后重新输入；满足后走「澄清 → 草案 → 证据 → 校验 → 输出」多步流程。规划节点内设计并落地 QP→召回→排序→规则过滤→证据组织 能力流水线：QP 输出召回查询与约束，经 Provider（Search/Map/Weather/Review）召回候选（POI/酒店/活动），可解释排序（偏好、预算、通勤、评分等）、规则过滤（超预算/时间冲突/闭馆等）、Evidence Builder 组织 evidence+refs（provider/url/fetched_at/attribution），产出符合 itinerary v1 契约的结构化行程（trip_profile、按天 slots、budget_summary、evidence 引用、预算分项、风险分级、备选项）并支持证据展示与 JSON/Markdown 导出。会话以 conversation_id 维系，会话级行程状态 MySQL 持久化。后端以 LangGraph 为唯一控制面，SSE 统一 envelope 回传阶段事件，保证阶段可见与失败回退。行程 Schema 带 schema_version 与 P0/P1 分级，P0 缺失仅回澄清，P1 降级写 assumptions。支持 Edit Day N 局部重规划（影响域含邻接时段）、revision/diff 与 request_id 幂等、changed_days 阈值确认与回滚。用户、会话、行程、revision、diff、evidence 存 MySQL，Redis 语义缓存。前端 Vue 3 + TypeScript，主工作台 TravelPlanner.vue。技术栈：Python、FastAPI、LangGraph、Redis、Vue 3、TypeScript。

## 四、分条版（可拆成简历 bullet）

- **定位与目标**：从零设计并实现旅行规划智能体，前后端分离，支持多轮对话，输出可编辑的 itinerary v1 行程与只读证据，不做交易闭环。
- **多轮对话**：以 conversation_id 维系会话，会话级行程状态持久化；澄清分支下用户可多轮补充（/travel/resume 续答）或 reset 清空，满足约束后进入主规划流程。
- **意图与 QP**：QP 做意图识别（create/edit/qa/reset）与约束抽取，硬门槛（目的地、天数、预算）缺失时由澄清服务判定并追问，补齐后再生成行程。
- **流程与流式**：LangGraph 为唯一控制面，编排「澄清→草案→证据→校验→输出」；SSE 统一 envelope 回传阶段事件，阶段可见、失败可回退。
- **检索与排序流水线**：规划节点内设计并落地 QP→召回→排序→规则过滤→证据组织。QP 输出召回查询与约束；经 Provider（Search/Map/Weather/Review）召回候选（POI/酒店/活动）；可解释排序（偏好匹配、预算、通勤、评分/热度、证据质量）；规则过滤（超预算、时间冲突、闭馆/停运、距离过远等）；Evidence Builder 组织 evidence+refs（provider/url/fetched_at/attribution），支撑可解释行程与证据展示。
- **行程契约与产品效果**：itinerary v1 含 schema_version、revision_id、trip_profile、days/slots、budget_summary、evidence、validation；P0 缺失仅回澄清，P1 降级写 assumptions。产品侧支持预算分项、风险分级、备选项、证据展示、JSON/Markdown 导出。
- **Edit Day N 与版本**：Edit Day N 局部重规划（影响域含邻接时段），revision/diff 记录改动，request_id 幂等，changed_days 阈值确认与回滚，降低改一处牵动全局的风险。
- **Provider 与数据**：地图/路线/POI 等通过 Provider 抽象（Search/Map/Weather/Review）封装，不绑定厂商；MySQL 存用户/会话/行程/revision/diff/evidence，Redis 语义缓存。

技术栈：Python、FastAPI、LangGraph、Redis、Vue 3、TypeScript。

## 六、方案与实现（对齐 design.md，面试可说明）

- **设计文档中的完整方案**：多轮对话与澄清、意图与 QP；主流程「澄清→草案→证据→校验→输出」；规划节点内能力流水线 QP→召回→排序→规则过滤→证据组织（design 5.3）；Provider 抽象（Search/Map/Weather/Review）；itinerary v1 契约与 P0/P1 降级；Edit Day N 与 revision/diff；预算分项、风险分级、备选项、证据展示、导出。上述方案在 design.md 中完整定义，与产品最终要呈现的效果一致。
- **当前已实现**：单次输入生成结构化行程草案；以 Edit Day N 为核心的局部调整；revision/diff 与 SSE 事件；澄清门槛与 QP 意图路由；itinerary v1 契约与 P0/P1 降级；会话态 MySQL 持久化；Provider 抽象层与 schema 预留。
- **目标态/渐进落地**：召回→排序→规则过滤→证据组织 在节点内完整打通并接入 travel 主链；同一会话内连续修改与 patch 编辑、前后端联动刷新。不推翻现有 LangGraph 主控与 itinerary v1 契约，按 M2/M3 里程碑逐步到位。

简历描述已包含「设计并落地」的完整方案，面试时可说明：方案按 design 与产品目标设计完整，主流程与契约已实现，检索重排证据链路在规划节点内按 design 5.3 推进落地。
