## 二、后端：LLM 生成真实行程（代码分析）

### 2.1 目标

不再只输出「模板占位行程」，而是用大模型按用户输入生成**具体地点、交通、费用**等，并保证失败时能回退到模板。

### 2.2 Prompt 设计（`llm_backend/app/domain/travel/draft_prompts.py`）

**系统提示 `TRAVEL_DRAFT_SYSTEM_PROMPT`**（约第 3–15 行）：

- 角色：专业旅行行程规划师。
- 约束：每天 3 个时段（上午/下午/晚上）、`place` 为真实地名、`transit` 写交通建议、`cost_breakdown` 在总预算内、有偏好则优先相关活动、每天有 `theme`。
- 输出形式：**纯 JSON**，不要 markdown 或解释文字。

**用户提示模板 `TRAVEL_DRAFT_USER_PROMPT_TEMPLATE`**（约第 17–91 行）：

- 占位符：`{destination_city}`, `{days}`, `{budget}`, `{traveler_type}`, `{preferences}`, `{pace}`，由调用方从 QP 解析结果填入。
- 模板里直接写好了目标 JSON 结构示例（`days` → `theme` / `slots` → `slot` / `activity` / `place` / `transit` / `cost_breakdown`，以及 `budget_summary`），让模型「照着填」。

这样做的目的：**用 prompt 约束输出形状**，方便后面用固定 schema 解析，减少解析失败。

### 2.3 图节点与 LLM 调用（`llm_backend/app/lg_agent/travel_draft_graph.py`）

**状态定义**（约第 37–45 行）：

```python
class TravelDraftInput(TypedDict):
    query: str

class TravelDraftState(TravelDraftInput):
    final_itinerary: dict | None   # 最终行程 JSON
    explanation: str | None         # 给用户看的说明
    final_text: str | None          # 缺 P0 时的澄清文案
```

**LLM 实例选择 `_get_llm()`**（约第 48–61 行）：

- 读 `settings.AGENT_SERVICE`：若为 `ServiceType.DEEPSEEK` 则 `ChatDeepSeek(api_key=..., model_name=...)`，否则用 `ChatOllama`。
- 即：**同一套图逻辑，通过配置切换 DeepSeek / Ollama**，无需改业务代码。

**模板降级 `_build_template_itinerary()`**（约第 64–89 行）：

- 当 LLM 不可用时调用。
- 用 `draft_builder.build_slots(i)` 生成占位 slot，拼成 `ItineraryDay` 列表，再拼成 `ItineraryV1`，并把 `assumptions` 写进 `itinerary.validation.assumptions`。
- 这样前端拿到的仍是合法 ItineraryV1，只是内容为占位。

**LLM 输出解析 `_parse_llm_itinerary()`**（约第 92–176 行）：

- 入参：LLM 原始字符串 `raw`，以及从 QP 得到的 `destination`、`days_count`、`total_budget`、`traveler_type`、`preferences`、`assumptions`。
- 预处理：若 `raw` 以 \`\`\` 开头则去掉首行和结尾的 \`\`\`，只保留中间 JSON 字符串（第 101–106 行）。
- 解析 `days`：遍历 `data["days"]`，对每个 slot 取 `cost_breakdown` 转成 `CostBreakdown`，再构造 `ItinerarySlot(place=..., transit=..., cost_breakdown=...)`（第 111–137 行）。
- 若 `days` 为空，则用 `build_slots` 生成占位天并往 `assumptions` 里追加说明（第 139–145 行）。
- 解析 `budget_summary`：取 `total_estimate`、`uncertainty_note`、`by_category` 填到 `BudgetSummary`（第 147–158 行）。
- 最后用 `TripProfile`、`days`、`budget_summary` 等组装成 `ItineraryV1`，并把传入的 `assumptions` 写进 `validation.assumptions`（第 160–175 行）。

**主节点 `generate_travel_draft()`**（约第 179–270 行）：

1. **QP 抽取**（第 180–183 行）：`extract_days` / `extract_budget` / `extract_destination` / `extract_traveler_type` 从 `query` 里取结构化信息。
2. **P0 校验**（第 185–201 行）：缺目的地、天数、预算任一时，直接返回 `final_text`（澄清文案），不调 LLM。
3. **偏好与节奏**（第 207–212 行）：从 `QP_RULES.preference_keywords` 和 `pace_keywords` 里用 `query` 匹配出 `preferences` 和 `pace`，传给 prompt。
4. **调用 LLM**（第 216–232 行）：
   - `_get_llm()` 拿到实例；
   - `TRAVEL_DRAFT_USER_PROMPT_TEMPLATE.format(...)` 填好用户 prompt；
   - `messages = [system, user]`，`await llm.ainvoke(messages)`；
   - 用 `_parse_llm_itinerary(response.content, ...)` 得到 `ItineraryV1`。
5. **异常降级**（第 252–264 行）：`except Exception` 里打日志，往 `assumptions` 追加「LLM 生成失败，已降级为模板」，再调用 `_build_template_itinerary(...)`，并生成说明文案。
6. **统一返回**（第 266–270 行）：`return { "final_itinerary": itinerary.model_dump(mode="json"), "explanation": ..., "final_text": None }`。上游会把 `final_itinerary` 通过 SSE 的 `final_itinerary` 事件发给前端。

**图结构**（第 273–278 行）：单节点图 `START → generate_travel_draft → END`，没有分支；P0 缺失时在节点内部直接 return，不建额外节点。

## 三、前端：Mindtrip 风格与数据对接（代码分析）

### 3.1 目标

把后端返回的 ItineraryV1（含 place、transit、cost_breakdown、theme、budget_summary）在界面上完整、美观地展示，并采用双栏 + 卡片 + 预算可视化的 Mindtrip 风格。

### 3.2 数据契约：TypeScript 接口（`frontend/DsAgentChat_web/src/views/TravelPlanner.vue`）

约第 224–302 行定义了与后端 ItineraryV1 对齐的接口，便于类型安全和后续扩展：

- **ItinerarySlot**：`slot`, `activity`, `place?`, `transit?`, `cost_breakdown?`, `risk?`, `alternatives?`, `evidence_refs?`
- **ItineraryDay**：`day_index`, `date?`, `theme?`, `slots: ItinerarySlot[]`
- **TripProfile**：`destination_city`, `constraints?`（内含 `budget_range`, `traveler_type`, `preferences`）
- **BudgetSummary**：`total_estimate`, `uncertainty_note?`, `by_category?`（transport/hotel/tickets/food/other）
- **ItineraryResult**：`trip_profile`, `days`, `budget_summary`, `validation?` 等

这样前端拿到的 `envelope.payload.itinerary` 可以安全地断言为 `ItineraryResult` 使用。

### 3.3 与后端的对接：SSE 与回调（`TravelPlanner.vue` 的 `submitQuery`）

约第 396–435 行：

- 调用 `ApiService.travelQueryStream({ query, userId, conversationId }, { ... })`。
- 在 `onFinalItinerary` 回调里：  
  `itinerary.value = envelope.payload.itinerary as ItineraryResult`，  
  `explanation.value = envelope.payload.explanation`，  
  并执行 `scrollToResult()`。
- 其他回调：`onStageStart` / `onStageProgress` → 进入 clarifying；`onFinalText` → 纯文案结果；`onError` → 错误提示；`onTextFallback` → 兜底文案。

也就是说：**后端通过 SSE 的 `final_itinerary` 事件把 `itinerary.model_dump(mode="json")` 推到前端，前端用同一套 ItineraryResult 类型渲染。**

### 3.4 左侧面板：输入、状态、旅行概览、预算

- **输入区**（约第 19–51 行）：`textarea` 绑定 `query`，按钮「开始规划」触发 `submitQuery`，「重置」触发 `resetPlanner`。`phase` 为 planning 时显示 loading 图标。
- **状态区**（约第 53–63 行）：根据 `phase` 显示不同 badge（正在生成草案 / 需要补充信息 / 草案已生成 / 请求失败），以及 `clarificationMessage`、`errorText`、`finalText`、`explanation`。
- **旅行概览**（约第 65–89 行）：`v-if="itinerary"` 时展示；`itinerary.trip_profile.destination_city` 作标题；标签用 `constraints.traveler_type`、天数、`constraints.budget_range`、`constraints.preferences` 数组。
- **预算卡片**（约第 91–115 行）：`itinerary.budget_summary` 存在时展示；总预算用 `formatNumber(itinerary.budget_summary.total_estimate)`；下方用计算属性 `budgetCategories` 生成带颜色的分类条。

**`budgetCategories` 计算属性**（约第 345–364 行）：

- 从 `itinerary.budget_summary.by_category` 取出 transport/hotel/tickets/food/other。
- 过滤掉无值或 ≤0 的项，再算每项占比 `percent`，并配上颜色和中文标签（交通/住宿/门票/餐饮/其他）。
- 这样模板里 `v-for="(cat, ci) in budgetCategories"` 就能画出分类条和金额。

### 3.5 右侧面板：空状态、加载、行程时间线

- **空状态**（约第 125–138 行）：`!itinerary && phase === 'idle'` 时显示插画和「你的行程将在这里呈现」等文案。
- **加载状态**（约第 140–145 行）：`!itinerary && phase === 'planning'` 时显示转圈和「正在为你规划行程…」。
- **行程时间线**（约第 157–214 行）：`v-if="itinerary"`，外层 `v-for="(day, dayIdx) in itinerary.days"`。
  - **天标题**：`day.day_index` 显示为「第 N 天」，`day.theme` 显示为副标题。
  - **时段列表**：`v-for="(slot, si) in day.slots"`，每个 slot 一个 `.tl-item`（时间线圆点 + 竖线 + 卡片）。
  - **卡片内容**：
    - `slot.slot`（上午/下午/晚上）；
    - `slot.activity` 作标题；
    - 若有 `slot.place` 则带图钉图标显示；
    - 若有 `slot.transit` 则带路线图标显示；
    - 若有 `getSlotCost(slot)` 则在卡片底部显示金额标签。

**`getSlotCost(slot)`**（约第 370–375 行）：把 `slot.cost_breakdown` 里 transport/hotel/tickets/food/other 相加，若大于 0 则返回，否则返回 null，用于决定是否显示「¥xxx」标签。

### 3.6 样式与动效（同一文件 `<style scoped>`）

- 设计 token：深色背景、青蓝主色、琥珀色费用、边框与悬停态（约 440 行起）。
- 左侧 380px 固定宽，右侧 flex；768px 以下改为上下堆叠。
- 卡片使用 `.fade-in` / `.slide-up`，预算条用 `animation: barScale`，时间线圆点与竖线在 `.tl-rail` 内。

## 四、端到端数据流（串起来看）

1. 用户在前端输入「京都 3天 预算5000 情侣」，点击「开始规划」。
2. `submitQuery` 调用 `ApiService.travelQueryStream`，后端收到 `/api/travel/query`，进入 LangGraph 的 `generate_travel_draft`。
3. 后端用 `extract_*` 得到 destination=京都、days=3、budget=5000、traveler_type=情侣等，拼好 prompt，调用 DeepSeek；DeepSeek 返回一段 JSON。
4. `_parse_llm_itinerary` 去掉 markdown 后解析成 `ItineraryV1`，再 `model_dump(mode="json")`，通过 SSE 的 `final_itinerary` 事件推给前端。
5. 前端在 `onFinalItinerary` 里把 `envelope.payload.itinerary` 赋给 `itinerary.value`，模板根据 `itinerary.days`、`itinerary.trip_profile`、`itinerary.budget_summary` 渲染左侧概览与预算、右侧按天按 slot 的卡片（place、transit、getSlotCost(slot) 等）。

这样，**从用户输入到界面展示的整条链路**，以及**每段逻辑在代码中的位置**，就都对应上了。
